# Astro UI 骨架改造方案

本文档用于定义 `/Users/snow/notes/code/projects/astro-i18n-starter` 下一轮前端骨架改造的目标、依据和实施顺序。

这不是泛泛的“页面优化建议”列表，而是把当前站点已经暴露的问题、适合借鉴的设计 skill、以及后续真正要落地的结构方案放到同一个文档里，作为后续改造 `Header`、`Footer`、首页骨架和移动端适配的统一基线。

## 1. 本轮改造目标

当前项目已经有可运行的页面和一套基础 token / UI primitives，但站点骨架层仍然有明显问题：

- 顶部导航布局失衡，品牌区、主导航、操作区被硬塞进一个过大的容器里。
- 桌面端 header 缺少清晰的信息优先级，导航换行后视觉非常乱。
- 移动端没有真正成立的导航策略，目前只是靠 `wrap` 和 `width: 100%` 被动挤压。
- footer 信息组织仍然是桌面优先，窄屏下阅读顺序和点击路径都不理想。
- 全局样式把“设计 token、骨架布局、页面组件、响应式补丁”混在一个大文件里，后续继续扩页面会越来越难控。

因此，本轮改造目标不是只修几个样式 bug，而是完成下面四件事：

1. 重建站点级骨架：header / footer / shell / section 的结构与节奏。
2. 明确移动端导航方案，而不是继续让桌面布局硬缩到手机上。
3. 把页面层视觉问题回收到稳定的布局规则和 token 上。
4. 为后续 Dashboard、Admin、Pricing、Blog 等页面统一出同一套外壳语言。

## 2. 当前页面问题归纳

基于当前代码和你给的页面截图，问题主要集中在骨架层，而不是单个组件本身。

### 2.1 Header 的核心问题

- 当前 header 把品牌区、主导航、语言切换、Dashboard、Login 全部塞进同一层 `grid`，信息密度过高。
- 导航项目较多时直接在容器里换行，导致第二行出现 `Couple Generator / Pricing / Features / Blog / About` 这种无节奏断裂。
- 品牌区文案过长，而且与主导航没有清楚的视觉层级关系，导致左侧块像一个独立卡片，和中间导航像两个系统。
- 语言选择器、Dashboard、Login 在右侧形成第三组视觉中心，使 header 一次出现三个主焦点。
- 当前 sticky header 本身透明、内部卡片体积过大，压住首页 hero 之后显得上重下轻。

### 2.2 Footer 的核心问题

- footer 目前是“营销文案 + CTA + 链接 + 法务说明”的混合体，但没有清晰拆层。
- 窄屏下 `footer-grid` 和 `footer-meta-wrap` 会堆叠，但阅读顺序仍偏桌面思维，不够自然。
- footer 链接组没有按用户任务拆分，只是平铺页面入口，和 header 有较高重复。
- footer CTA 与页尾 meta 在视觉语气上过于接近，导致页尾缺少明确收束。

### 2.3 首页骨架的核心问题

- hero 上方的站点 header 本身过大，抢走了首页首屏应有的聚焦点。
- hero 左右两栏之间的视觉重心不均衡，右侧工具卡没有形成强辅助，而是与主标题抢主角。
- 指标卡、CTA、工具入口、FAQ 都在使用相近的卡片语言，但层级差异不够明显。
- 页面内容块已经不少，但 section 之间的垂直节奏还不够稳定，读起来有“一个个盒子堆上去”的感觉。

### 2.4 样式组织问题

- `src/styles/global.css` 同时承担 reset、token 消费、布局规则、组件规则、响应式补丁，职责过重。
- header/footer 的规则没有被抽成独立骨架层，后续改动容易影响页面其他卡片和按钮。
- 一些布局修复是通过媒体查询末尾补丁完成的，这种方式短期有效，长期会把样式复杂度推高。

## 3. UI Skills 调研结果

为了避免这轮改造继续停留在“凭感觉修样式”，我先筛了一轮 `https://www.ui-skills.com/skills/` 里的 skill，重点看了那些真正能帮助站点骨架、信息层级、可访问性和交互节奏的内容。

不是所有 skill 都适合这个项目。对当前 Astro 站点最有帮助的是下面这些。

### 3.1 `ibelick/baseline-ui`

价值：适合作为“最低 UI 质量基线”。

它强调的方向和当前项目高度相关：

- 避免 AI 式拼接界面。
- 强制建立清晰的排版层级。
- 不要混乱地堆交互原语。
- 固定元素要考虑安全区域和焦点顺序。
- 动画必须克制，且只在明确需要时使用。

对本项目的具体启发：

- header 和 footer 不能继续只是“把现有按钮摆进去”，必须先明确哪部分是骨架、哪部分是动作、哪部分是辅助信息。
- 移动端菜单必须是明确的交互组件，而不是桌面导航自动换行。
- 骨架改造阶段要优先解决结构问题，再决定是否加动效。

### 3.2 `anthropics/frontend-design`

价值：适合作为“避免做出模板味太重的中庸页面”的设计约束。

它强调的是：

- 前端设计不能只停留在组件堆叠。
- 页面要有明确视觉方向和主次关系。
- 需要做出有辨识度但仍可生产落地的界面。

对本项目的具体启发：

- 首页 header 不应该像后台导航条直接搬到营销页。
- hero、工具入口、指标卡之间要有明确角色分工，而不是全都做成接近的浅色卡片。
- 站点骨架要有“产品感”，而不是继续停留在 starter 模板状态。

### 3.3 `pbakaus/layout`

价值：直接对应当前最严重的问题，即 spacing、composition、hierarchy rhythm。

这个 skill 的描述非常贴合现在的现状：当界面显得拥挤、扁平、错位时，优先修布局节奏，而不是局部补色。

对本项目的具体启发：

- header 需要从“三栏塞满”改成更稳定的组合关系。
- 品牌区宽度、主导航密度、操作区按钮数量要重新分配。
- 首页首屏和后续 section 的上下间距需要形成稳定节拍。
- footer 要从单纯堆块，改成“主信息区 + 次导航区 + 法务区”的分层结构。

### 3.4 `zeke/swiss-design`

价值：适合作为站点骨架和排版系统的审美方向参考，不是要把页面做成纯瑞士风，而是借它的网格纪律和层级克制。

它最有价值的点在于：

- 网格纪律优先于装饰。
- 强调标题、说明、辅助文本的节奏差。
- 用更少但更明确的视觉手段建立秩序。

对本项目的具体启发：

- header 里的品牌文案应当压缩并分层，不该把 tagline 整段塞在导航主干里。
- 首页 section heading 应该更统一，避免每块都像不同页面拼接。
- footer 链接组可以做得更像“信息索引”，而不是一串重复入口。

### 3.5 `vercel-labs/web-design-guidelines`

价值：适合作为改造后的复查标准，而不是视觉灵感来源。

它本质上要求：

- 交互语义清楚。
- 键盘可达。
- 焦点状态明确。
- 固定元素和导航行为不能影响可用性。

对本项目的具体启发：

- 移动端菜单如果要引入抽屉、折叠或下拉，必须先保证语义和键盘行为是对的。
- 语言切换器和导航入口在移动端不能变成难以聚焦或难以点击的小目标。
- sticky header 不能遮挡核心内容的可访问路径。

### 3.6 `ibelick/fixing-accessibility`

价值：适合约束移动端 header 改造时的无障碍底线。

对本项目尤其重要的点：

- 所有交互控件必须有清晰名称。
- 自定义菜单不能只做视觉折叠，必须考虑键盘和焦点。
- 表单和切换控件不能只依赖颜色表达状态。

对本轮的现实意义是：后续只要引入移动端菜单按钮、展开层、关闭按钮，就必须按这个标准收口。

### 3.7 `wshobson/interaction-design`

价值：用于定义“该加什么动效，不该加什么动效”。

它强调：

- 微交互要服务理解，不是为了炫。
- 只优先使用 `transform` / `opacity` 一类高性能动画。
- 要尊重 `prefers-reduced-motion`。

对本项目的具体启发：

- 移动端菜单可以有轻量展开过渡，但不能做大面积模糊、位移动画叠加。
- sticky header 的出现、CTA hover、卡片反馈都应是弱动效，而不是继续堆阴影和 blur。

### 3.8 `jakubkrehel/make-interfaces-feel-better`

价值：适合做最后一层精修，而不是作为主结构方案。

它比较适合解决这些收尾问题：

- 圆角层级不一致。
- 数字与文本的排版质感不足。
- 卡片、图片、按钮的细节过硬或过钝。
- 交互反馈有但不够精致。

对本项目的具体启发：

- 首页统计卡和按钮的细节可以后续用它来统一精修。
- 但前提是先把骨架结构做对，否则“打磨感”只会掩盖结构问题。

## 4. 最终采用的改造原则

综合上面的调研结果，这个项目的骨架改造不采用单一 skill 的完整风格，而采用下面这套组合原则：

- 用 `baseline-ui` 管底线：结构先行，交互要有语义，动画克制。
- 用 `frontend-design` 管方向：界面要摆脱 starter 模板感，形成明确品牌骨架。
- 用 `layout` 管秩序：先重做空间分配、层级节奏和页面构图。
- 用 `swiss-design` 管排版：减少噪音，提升信息层级和索引感。
- 用 `web-design-guidelines` + `fixing-accessibility` 做验收：尤其针对移动端菜单和 sticky 导航。
- 用 `interaction-design` + `make-interfaces-feel-better` 做收尾：只给真正需要的地方补微交互和质感。

换句话说，这轮改造优先级是：

1. 结构正确
2. 层级清楚
3. 响应式成立
4. 可访问性成立
5. 质感和微交互最后补齐

## 5. 针对本项目的骨架改造方案

### 5.1 Header 改造方向

当前 header 不再适合继续做“品牌区 + 全量导航 + 全量操作”同屏并列，建议改成两套明确策略。

#### 桌面端

- 左侧保留简化版品牌识别：logo + 产品名。
- 中间保留一层主导航，但只承载真正高频入口。
- 右侧保留语言切换和核心动作，避免同时挂太多按钮。
- 次级信息不要继续放进 header 主干，可下沉到页面内或 footer。

建议的桌面信息优先级：

1. `Home`
2. `Photo to Chibi`
3. `Avatar Maker`
4. `Couple Generator`
5. `Pricing`
6. 其余如 `Features` / `Blog` / `About` 进入次级入口或更克制的展示方式

这不是说必须隐藏这些页面，而是 header 不应该承担整个 sitemap。

#### 移动端

- 采用明确的菜单按钮展开方案。
- 顶层只保留：品牌、菜单按钮、一个高优先动作。
- 展开层内再放导航、语言切换、Dashboard、Login。
- 移动端菜单必须具备清晰的层级顺序和足够的点击面积。

移动端不能再依赖：

- `flex-wrap`
- `grid` 自然掉行
- 按钮强制 `width: 100%` 兜底

这些只能作为补丁，不能作为导航策略。

### 5.2 Footer 改造方向

footer 建议从现在的“两个大块 + 一行 meta”重构为三层：

1. 品牌与价值主张层
2. 链接索引层
3. 法务与版权层

推荐结构：

- 左侧：品牌说明、简短产品描述
- 右侧：一组明确 CTA，但不要和 header 按钮气质完全重复
- 下层：按任务拆分链接，如 `Product`、`Company`、`Support`
- 最底层：版权、rights、可能的政策入口

这样 footer 才是真正的收尾层，而不是另一个重复 header。

### 5.3 首页骨架改造方向

首页建议沿着“一个主叙事 + 两个辅助层”的结构整理。

推荐顺序：

1. 首屏 hero：一句强标题 + 一句说明 + 一个主 CTA + 一个次 CTA
2. 可信度 / 结果感：指标或示例，不要与 hero 抢主导
3. 工具入口区：作为产品导航，而不是另一块 hero
4. 信息说明区：特性、使用建议、FAQ 依次展开

关键点：

- 首屏只允许一个主视觉中心。
- 右侧辅助卡不能再像第二个主标题区。
- 统计卡与工具卡需要拉开语气，不要全部同一个卡片语言。

### 5.4 样式文件组织改造方向

建议把当前样式按职责拆层，而不是继续把所有规则堆进 `global.css`。

建议拆分方向：

- `theme.css`：token 与设计变量
- `global.css`：reset、基础元素、全站共性排版
- `layout.css`：shell、header、footer、section、page frame
- `components.css`：卡片、导航项、工具条、统计卡等通用组件皮肤

如果暂时不想拆成太多文件，至少也要先把“站点骨架规则”从通用卡片规则中分离出来。

## 6. 具体实施顺序

建议按下面顺序执行，而不是同时平推所有页面。

### 第一阶段：站点骨架纠偏

- 重构 `Header.astro`
- 重构 `Footer.astro`
- 把 header/footer 的样式从杂糅规则里独立出来
- 先完成移动端导航策略

这是第一优先级，因为当前最显著的问题就在这里。

### 第二阶段：首页首屏与 section 节奏重排

- 调整 hero 的主次关系
- 统一统计卡与工具卡的角色差异
- 重新校准 section 间距与内容顺序

### 第三阶段：可访问性与交互收口

- 检查菜单按钮、展开层、语言切换器的语义
- 检查键盘焦点顺序
- 检查 sticky header 的阅读与滚动体验
- 补轻量而克制的状态动效

### 第四阶段：视觉精修

- 统一圆角层级
- 统一标题与正文的字阶和行长
- 微调阴影、边框、描边和 hover 反馈

## 7. 本轮改造的明确边界

为了避免范围失控，这轮先不做下面这些事：

- 不先重做整套品牌视觉识别。
- 不先引入复杂动画系统。
- 不先扩充更多 UI 组件种类。
- 不先同时重构 Dashboard / Admin 全部页面。

本轮只聚焦：

- 站点骨架
- 首页首屏节奏
- 移动端 header / footer 成立
- 样式组织变得可继续维护

## 8. 最终判断

当前项目不是“缺几个 media query”，而是骨架策略本身还停留在 starter 拼装阶段。

这轮改造的正确方向不是继续补丁式修 CSS，而是：

- 先按 `layout` 思路重做信息分配和空间秩序。
- 用 `baseline-ui` 和 `web-design-guidelines` 守住结构与可访问性底线。
- 用 `frontend-design` 和 `swiss-design` 把页面从模板感拉出来。
- 最后再用 `interaction-design` 和 `make-interfaces-feel-better` 做细节精修。

后续真正开始改代码时，应以这份文档作为约束，不再把 header/footer 当成普通组件局部修补，而是作为全站骨架系统一起重构。
